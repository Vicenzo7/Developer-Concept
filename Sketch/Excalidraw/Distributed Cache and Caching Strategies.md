What is Caching?
-> Caching is a technique to store frequently used data in a fast access memory rather than
accessing data every time from slow access memory.
-> This makes our system very fast.
-> Helps us to reduce latency.
-> Also helps us to achieve fault tolerance.

2. CDN (used to store static data)

Caching Strategies ^UbzIsZcc

1. Cache Aside

-> First data is checked is cache
-> Cache Hit data is returned to the user
-> Cache Miss, application fetches the data from DB and stores it in cache and returns data
   to user

Pros
1. Good Approach for heavy read application.
2. Even if Cache is down, the request will not fail as it will fetch data from DB.
3. Cache Document Data Structure could be different that what is stored in DB.

Cons
1. For few data reads, there will always be cache miss first.(To resolve this we can generally
   pre-heat the cache first)
2. Data can be inconsistent ^XE7kdhj0

2. Read through cache

-> Data is first checked in cache.
-> Cache hit data is sent back to the user
-> Cache miss, cache itself makes a call to DB to retrieve the data, stores it in cache
and return it to the user.

Pros
1. Good Approach for heavy read operation
2. Logic of fetching the data from DB and updating the cache, is separated from application.

Cons
1. For few data reads, there will always be cache miss first.(To resolve this we can generally
   pre-heat the cache first)
2. Data can be inconsistent
3. Cache Document Structure should be same as table. ^6yxmJdyG

3. Write Around cache

-> Data is written directly to the DB.
-> Invalidates the data in the cache


Pros
1. Good Approach for heavy read operation
2. Resolves inconsistency problem between cache and DB.

Cons
1. For few data reads, there will always be cache miss first.(To resolve this we can generally
   pre-heat the cache first)
2. If DB is down write operation will fail ^ASoNaipK

4. Write Through cache

-> First write data to the cache.
-> Then in synchronously write data to the DB


Pros
1. Cache and DB always remain consistent.
2. Resolves inconsistency problem between cache and DB.
3. Cache Hit chance increases a lot.

Cons
1. Alone it is not useful it will increase the latency, hence always used with 
read through or cache aside cache.
2. 2 Phase commit needed to be supported with this, to maintain the transactional property.
3. If DB down write operation will fail. ^jloGtnLT

5. Write Behind (or Back) cache

-> First write data into the cache
-> Then in asynchronously write data to the DB 

Pros

1. Good for write heavy application.
2. Improves write operation latency, as writing to the DB happens asynchronously.
3. Cache hit chances increases a lot.
4. Gives much better performance when used with read through cache.

Cons
1. Even when DB fails write operation ill still work.
2. If data is removed from cache and write operation in DB still does not happen, then 
there is a chance of an issue. This is handled by keeping TAT (Turn Around Time) of the cache little higher like
2 days ^FPvPbbuC

Cache Eviction Policies are 
1. LRU
2. LFU
3. FIFO ^QXmBM7X0
